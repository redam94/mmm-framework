# CLAUDE.md - MMM Framework Developer Guide

## Project Overview

**MMM Framework** is a production-ready Bayesian Marketing Mix Modeling framework extending PyMC-Marketing with advanced capabilities for modeling marketing effectiveness. Emphasizes methodological rigor, genuine uncertainty quantification, and pre-specified analyses to reduce researcher degrees of freedom.

**Version**: 0.2.0
**Author**: Matthew Reda
**Python**: 3.12+

## Quick Commands

```bash

uv sync --group dev --group app

make fast_tests         # Skip slow tests (parallel)
make slow_tests         # Slow tests only (parallel)
```

## Code Style

- **Formatter**: Black (via `make format`)
- **Linter**: Ruff
- **Type hints**: Required (py.typed markers present)
- **Docstrings**: Google-style
- **Naming**: PascalCase classes, snake_case functions, UPPER_SNAKE constants

## Architecture Patterns

- **Builder Pattern**: Fluent configuration (e.g., `ModelConfigBuilder().with_kpi("sales").build()`)
- **Factory Functions**: Pre-configured builders (e.g., `awareness_mediator()`)
- **Lazy Imports**: mmm_extensions delays PyMC import until needed
- **Pydantic Models**: Type-safe configuration validation
- **Subpackage Organization**: Related modules grouped into subpackages (builders/, model/, transforms/, etc.)
- **DAG-based Configuration**: Frontend-driven model specification via dag_model_builder/

## Core Technologies

- **PyMC 6.0+** - Bayesian inference (lockstep with PyTensor 3.x + ArviZ 1.x — ArviZ containers are xarray DataTrees; route arviz calls through `utils/arviz_compat.py`)
- **NumPyro 0.19+** - Fast NUTS sampling
- **NutPie 0.16+** - Alternative fast NUTS sampler
- **Numba 0.63+** - JIT compilation for performance
- **FastAPI 0.124+** - REST API
- **Streamlit 1.52+** - Legacy web UI
- **React + TypeScript** - Modern web UI (Vite, TailwindCSS, Zustand)
- **Redis 7.1+ / ARQ 0.25+** - Async job queue

- **Pydantic 2.12+** - Data validation

## Common Development Tasks

### Add feature to BayesianMMM
1. Edit `src/mmm_framework/model/base.py`
2. Update results in `src/mmm_framework/model/results.py`
3. Update config in `src/mmm_framework/config/`
4. Add builder method in `src/mmm_framework/builders/model.py`
5. Write tests in `tests/test_model.py`

### Add new transform
1. Create transform function in `src/mmm_framework/transforms/`
2. Export in `src/mmm_framework/transforms/__init__.py`
3. Add tests in `tests/test_transforms.py`

### Add report section
1. Chart function in `reporting/charts/`
2. Extractor in `reporting/extractors/`
3. Section builder in `reporting/sections.py`
4. Helper in `reporting/helpers/`
5. Update `reporting/config.py`

### Add extended model
1. Config in `mmm_extensions/config.py`
2. Builder in `mmm_extensions/builders.py`
3. Components in `mmm_extensions/components/`
4. Model class in `mmm_extensions/models/`
5. Results in `mmm_extensions/results.py`

### Add API endpoint
1. Create route in `api/routes/`
2. Add schemas in `api/schemas.py`
3. Register in `api/main.py`
4. Add tests in `api/tests/`

### Add frontend page (React)
1. Create page component in `frontend/src/pages/`
2. Add API hooks in `frontend/src/api/hooks/`
3. Update store if needed in `frontend/src/stores/`
4. Add route in `frontend/src/App.tsx`

## API Usage Examples

```python
# Basic model fitting
from mmm_framework import BayesianMMM
from mmm_framework.builders import ModelConfigBuilder

config = ModelConfigBuilder().with_kpi("sales").build()
model = BayesianMMM(X_media, y, channel_names, config)
results = model.fit(draws=2000, tune=1000)

# Data loading
from mmm_framework import MFFLoader
loader = MFFLoader(config=mff_config)
data = loader.load("data.csv")

# Report generation
from mmm_framework.reporting import MMMReportGenerator, ReportConfig
generator = MMMReportGenerator()
html = generator.generate_report(results, config)

# Model serialization
from mmm_framework.serialization import MMMSerializer
MMMSerializer.save(model, "model.pkl")
model = MMMSerializer.load("model.pkl")

# Analysis
from mmm_framework.analysis import MMMAnalyzer
analyzer = MMMAnalyzer(model)
contributions = analyzer.compute_counterfactual_contributions()
marginal = analyzer.compute_marginal_contributions(spend_increase_pct=10)
```

## Troubleshooting

| Issue | Solution |
|-------|----------|
| Redis connection fails | Run `redis-server` in separate terminal |
| Worker not processing | Start ARQ: `cd api && arq worker.WorkerSettings` |
| Model fitting slow | Use numpyro sampler: `model.fit(nuts_sampler="numpyro")` |
| Approximate / fast fit for model checking | `BayesianMMM.fit(method=...)` runs an *approximate* posterior in seconds to spot problems (bad priors, broken geometry, pathological saturation/adstock) before paying for NUTS. `method`: `"map"` (point estimate), `"advi"`/`"fullrank_advi"` (variational; `find_MAP`/`pm.fit` from core PyMC — no extra deps), or `"pathfinder"` (works OUT OF THE BOX: `pymc-extras>=0.12,<0.13` is a declared core dependency, tracking the pymc 6 baseline — 2026-07-06; it was first declared 2026-07-05 as `>=0.6,<0.7` against the old pymc 5.26 stack. Constant Deterministics broke pymc-extras' trace conversion, so `model/base.py::_anchored_det` zero-weight-anchors any parameter-independent component Deterministic (national geo/product, seasonality-off, trend-none, no-controls, observed-data `y_obs_scaled` — observed RVs are ancestry BLOCKERS since conversion substitutes their data) on the intercept; param-dependent graphs untouched. Invariant pinned by `tests/test_approx_fit.py::test_no_parameter_independent_deterministics`). Returns a normal `MMMResults` with `approximate=True` and `diagnostics["fit_method"]`; the posterior is a drop-in for the NUTS trace (deterministics included, `predict`/reporting work) but R-hat/ESS are `None` and uncertainty is **not** calibrated — re-fit with NUTS before trusting intervals/decisions. Enum `FitMethod`; builder `.with_fit_method(...)`/`.map_fit()`/`.advi(full_rank=)`/`.pathfinder()` set `ModelConfig.fit_method` (default `nuts`). Agent: `fit_mmm_model(method=...)` or spec `inference.method`. Impl: the model-agnostic engine `model/base.py::run_approximate_fit(model, method, draws, seed)` (both `BayesianMMM._fit_approx` AND the DAG-routed **extension models** — `NestedMMM`/`MultivariateMMM`/`CombinedMMM` — call it, so `.fit(method="map"/"advi"/…)` now works on extensions too; `BaseExtendedMMM.fit` gained a `method` arg, `ModelResults.approximate` reads `diagnostics["approximate"]`, and `agents/fitting.build_and_fit` passes `method` through instead of the old `ValueError` "not available for extension models"). NB Pathfinder's constant-Deterministic caveat (`_anchored_det`) is only applied on the base graph; a bespoke extension graph with parameter-independent Deterministics may still need it for Pathfinder (MAP/ADVI unaffected). **Approximate ≠ full posterior is now surfaced end-to-end (2026-07-07):** (1) **reports flag it** — the extractors recomputed R-hat/ESS from the raw trace and DISCARDED the model's own flags, so a MAP/ADVI fit read as a calibrated NUTS posterior; `reporting/extractors/base.py::_merge_fit_provenance` now folds `approximate`/`fit_method` from `results.diagnostics` into `bundle.diagnostics` (nulls R-hat/ESS + re-derives `converged` → "not assessable", never green), called at both `bayesian.py`/`extended.py` diagnostics sites (needs `results` passed to the extractor — the agent/ReportBuilder paths do). Classic report gets a prominent `ExecutiveSummarySection._render_approximate_banner` ("Approximate fit (MAP) — uncertainty is not calibrated"); the augur CLIENT deck (no diagnostics section) gets `AugurHeadlineSection._caveat_banner` (approximate OR non-converged, plain-language); prefit `helpers/prefit.model_assumptions` Inference-plan row reads `model_config.fit_method` (describes MAP/ADVI, doesn't promise post-fit diagnostics; emitted even when `mc is None` for extended). (2) **artifacts self-describe** — `model/base.py::fit` writes the resolved method back to `self.model_config.fit_method` (was a stale constant "nuts" in `configs.json`); `build_and_fit`'s `model_run` records `model_class`+`model_kind`+`approximate` first-class (was `model_class` for extensions only); serializer `metadata.json` gains `fit_method`+`approximate`; `saved_model_settings` surfaces the model class + `settings_digest_markdown` a "Model:" line; `planning/history.compute_run_metrics` stamps `approximate`/`fit_method` so persisted ROI/CI history isn't mistaken for calibrated. `agents/fitting.build_model` sets `model_config.fit_method` from `spec.inference.method` so an UNFITTED model (prefit/prior-predictive) reflects the planned method. (3) **priors reflect the model to be fitted** — the consumed-paths registry is now DAG-aware: `unconsumed_prior_path` rejects ALL `priors.*` and `unconsumed_spec_path` rejects `media_prior_mode` for DAG-routed extension specs (their priors come from the causal DAG node configs, dropped at `_build_extension_model`), so `update_model_setting` TELLS the user the write won't apply instead of a false success (inspect actual extension priors via `prior_predictive_check`, which already builds the real extended model). (The formerly-deferred extension-prior settability + trend/seasonality/likelihood wiring is now SHIPPED — see the "Extension-model priors + trend/seasonality/likelihood" row.) Tests `tests/test_approx_fit.py`, `tests/test_extension_fit_path.py`, `tests/reporting/test_approximate_fit_reporting.py`. |
| Extension-model priors + trend/seasonality/likelihood (spec-settable) | The DAG-routed extension models (`NestedMMM`/`MultivariateMMM`/`CombinedMMM`) used to **discard** the spec's trend/seasonality/likelihood and expose NO settable priors (every prior came from the causal-DAG node-config defaults; the plain-model `priors.*` were silently dropped).
