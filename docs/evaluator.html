Maturity & API Stability

The framework is pre-1.0 and says so. Different modules have matured at different rates, and the changelog page maintains the authoritative tier list. In summary:

  • Stable — the core modeling surface: BayesianMMM, results classes, the fluent builders, Pydantic configs, the adstock/saturation/seasonality/trend transforms, and the MFF data loader.
  • Beta — the active surface: reporting, analysis, serialization, the FastAPI endpoints, the React web UI, the AI agent workspace, the measurement-loop planning engine, and the pre-fit EDA module.
  • Experimental — subject to change: the extension models (NestedMMM, MultivariateMMM, CombinedMMM), the DAG model builder, the Excel config workflow, and the legacy Streamlit UI.

Until 1.0, minor versions (0.x.0) may break experimental modules; stable APIs are preserved on a best-effort basis with deprecation warnings. After 1.0 the project commits to strict SemVer.

Roadmap to 1.0 — criteria, not dates

There is no promised ship date. 1.0 happens when specific criteria are met: stabilizing the Beta-tier APIs, and publishing to PyPI. Until then, pin a git commit for production use.

What's Measured vs. What's Claimed

The marketing for most MMM tools is assertion. This project's policy is that every quantitative claim on the docs site traces to a seeded, re-runnable notebook—so any number you read can be re-derived by re-baking the notebook, not taken on trust. The four proof assets an evaluator should weigh:

  • Pressure-testing scorecard. The framework is fit to 16 synthetic worlds with known causal ground truth, and every estimate is graded against that truth—including the 8 worlds where it fails silently. The recorded scorecard, with failures, is public: pressure-testing.
  • Forecast accuracy backtest. Rolling-origin out-of-sample accuracy is measured, not asserted: forecasting workflow — backtesting.
  • Runtime benchmark. A production-size national fit (156 weeks × 7 channels, NumPyro, 4 chains × 500 draws) measures ~17 s wall-clock on an Apple M3 laptop, including compilation. Source: nbs/runtime_benchmark.ipynb; full scaling table in the technical guide's Computational Scaling section. Absolute seconds are machine-specific; treat the ratios as the transferable part.
  • Real-data test. The framework is run on the Lydia Pinkham advertising series (1907–1960), the most-studied real dataset in advertising econometrics, and graded against decades of published estimates: nbs/pinkham_real_data.ipynb.

The methodology itself is documented in prose—the technical guide and the mathematics series—rather than living only in code, so it can be audited without reading the source.

Team & Sustainability

This framework is built and maintained by one person: Matthew Reda. That is the single largest risk in adopting it, and you should weigh it as such. If the maintainer is unavailable, there is no team to pick up the work. We are not going to spin that.

What genuinely mitigates the bus factor:

  • Apache-2.0 license. You can fork the entire codebase and continue without permission. Nothing is held back in a proprietary tier.
  • A test suite of 2,000+ passing tests that lets a successor (or your own team) modify the code with a safety net.
  • Pre-registered, seeded notebook evidence. Every measured claim on this site comes from a notebook that can be re-baked—the evidence base does not depend on the maintainer's memory.
  • A pressure-testing harness with a recorded scorecard that is re-recorded whenever framework defaults change, so regressions in methodology are visible, not silent.
  • Methodology stated in documentation, not hidden in code. The technical guide and math series describe what the model does and why, so the design intent survives independently of any individual.

If your procurement process requires a vendor with an organization behind it, this project does not currently meet that bar. If it requires software your own team can own, audit, and extend, the items above are designed for exactly that.

Security & Deployment

The framework is self-hosted: a Python library, a FastAPI backend with a Redis-backed job queue, and a React frontend that you run on your own infrastructure. There is no hosted SaaS offering, which means no client data leaves your environment by default.

The AI agent workspace introduces an LLM data flow that deserves separate scrutiny—which provider sees what, sandboxing of agent-executed code, and the configuration options (including fully local models via LM Studio). That is covered on the dedicated security page rather than duplicated here.

Data Requirements & Runtime

Input data uses the MFF (Master Flat File) long format, loaded and validated by MFFLoader. The practical floor and the recommended shape of the data—granularity, history length, channel coverage—are documented on the data requirements page. Short version: 104+ weeks of weekly data is the comfortable starting point.

Runtime is measured, not estimated: the reference production-size national fit completes in ~17 s on a laptop (see Computational Scaling), and geo-panel and extension models scale sub-linearly per row at that size. Model iteration speed is not a procurement risk here.

How to Evaluate This in 30 (or 120) Minutes

30 minutes — reading only

Read the pressure-testing scorecard first, including the failure column—a tool that publishes where it breaks is showing you its actual epistemics. Then the forecast backtest for out-of-sample honesty, and skim Aurora 05 to see the full workflow end to end.

120 minutes — hands on

Do the 30-minute pass, then clone the repository and follow the quickstart: install from source, fit a first model on synthetic data with known ground truth, and check the estimates against the answer key yourself. If you want to stress the evidence, re-run nbs/runtime_benchmark.ipynb on your own hardware and compare.

Known Limitations

⚠️ Read before depending

  • Not on PyPI. Source installs only; pin a git commit for anything production-facing.
  • Solo maintainer. No SLA, no commercial support, real bus-factor risk (mitigations above).
  • Pre-1.0 API churn. Beta and Experimental modules may change between minor versions; only the core modeling tier is treated as stable.
  • No SaaS offering. You operate the stack (FastAPI, Redis, worker, frontend) yourself.
  • Open methodological findings. The pressure-testing page records unresolved issues openly—e.g. a holdout-validation fallback bug for Hill-fit models. (The former headline finding—the legacy blend adstock underperforming the parametric path—was resolved in June 2026 by making parametric adstock the default.) See known strengths & weaknesses.

None of these are hidden in fine print elsewhere—this list is the fine print, on purpose. If any item is disqualifying for your context, better to know now.
