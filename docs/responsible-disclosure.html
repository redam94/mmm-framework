Responsible Disclosure

If you have found a security vulnerability in the MMM Framework, this page tells you how to report it, what is in scope, and what to expect in response. The framework is self-hosted, open-source software — there is no vendor-operated SaaS to attack, so every finding lands in code you (and we) can read.

How to Report

Use whichever channel you prefer. Private channels first, please — give us a chance to ship a fix before the details are public.

ChannelUse it for
GitHub Security AdvisoryPreferred. Private, threaded, and lets us collaborate on a fix and coordinate a CVE if warranted.
m.reda94@gmail.comEmail, if you cannot use GitHub. Encrypt or share a minimal reproduction; we will move to a private advisory.
Public GitHub issueOnly for low-severity, already-public, or configuration/hardening questions — never for an exploitable, unfixed vulnerability.

The machine-readable policy is published at /.well-known/security.txt per RFC 9116.

What Helps Us Triage Fast

  • The affected file and line, or the endpoint/route — every control in this project traces to a specific source location, so a file reference is the fastest path to a fix.
  • A minimal reproduction or proof-of-concept.
  • The deployment posture you tested under — single-user development (the default, no flags) or hosted multi-user (MMM_AGENT_HOSTED=1). The two postures have materially different threat models, so this changes severity.
  • Impact: what an attacker gains, and any pre-conditions (auth enabled? which kernel mode?).

Scope

In scope is the source in this repository: the modeling library, the FastAPI backend, the agent (LLM-calling, code-executing) subsystem, the kernel sandbox, authentication and tenant-isolation code, and the documentation site. Findings are most actionable when framed against the posture they affect.

Read the posture first

Before reporting, please read the Security & Deployment Posture page. Several behaviors are documented, intentional properties of the default single-user posture rather than vulnerabilities — for example, the default install has no login (MMM_AUTH_ENABLED defaults to off) and the default inprocess kernel runs agent code in the API process with no sandbox. These are stated plainly there. The hardened controls (multi-tenant isolation, the container kernel sandbox, denied egress) only engage when the corresponding flags are set, and the hosted profile refuses to start if its sandbox image/runtime is absent. A report that the default posture lacks a control that is documented as off-by-default is a hardening discussion, not a vulnerability.

Especially interesting

  • Tenant isolation bypass when MMM_AUTH_ENABLED=1 — cross-org read/write, IDOR, or a route missing its access dependency.
  • Sandbox escape from the container kernel when MMM_AGENT_KERNEL=container (or under the hosted profile): host secret/credential exfiltration, sibling-session workspace access, or egress where egress should be denied.
  • SSRF in the website brand-extraction fetch, including bypasses of the public-address check.
  • Authentication weaknesses in the JWT or password paths.
  • Path traversal / TOCTOU in file download serving, or capability leakage of content-addressed plot/table IDs.

Out of scope

  • Findings that require the operator to have explicitly disabled a documented safeguard (e.g. MMM_KERNEL_SCRUB_ENV=0, MMM_KERNEL_EGRESS=open).
  • The default single-user posture's documented lack of auth/sandbox (see the posture note above).
  • Vulnerabilities in third-party dependencies — please report those upstream; tell us so we can pin/patch.
  • Volumetric DoS against a self-hosted instance you control.

What to Expect

  • Acknowledgement as soon as we see the report (this is a small open-source project, so please allow a few days).
  • Triage against the relevant posture and confirmation of whether we can reproduce.
  • A fix or mitigation in the source, with the change referenced back to the report.
  • Coordinated disclosure — we will agree a timeline with you and credit you in the advisory and changelog if you wish.

Good-faith safe harbor

We will not pursue action against researchers who act in good faith: test only against your own instance, avoid privacy violations and data destruction, do not exfiltrate more data than needed to prove a finding, and give us a reasonable window before public disclosure. There is no paid bug-bounty program — this is volunteer-maintained open-source software — but credit is given gladly.
